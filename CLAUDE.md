# CLAUDE.md

Поведенческие и проектные гайдлайны для работы с IndiaHorizone — tech-enabled India concierge для русскоязычных клиентов.

---

## Принципы разработки

**Компромисс:** Эти принципы смещены в сторону осторожности, а не скорости. Для тривиальных задач используй здравый смысл.

### 1. Думай перед кодом

**Не предполагай. Не скрывай путаницу. Озвучивай компромиссы.**

- Явно формулируй допущения. Если не уверен — спроси.
- Если есть несколько интерпретаций — представь их все, не выбирай молча.
- Если есть более простой подход — скажи об этом.
- Если что-то неясно — остановись. Назови, что именно непонятно.

### 2. Простота прежде всего

**Минимум кода, который решает задачу. Ничего спекулятивного.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости», которую не просили.

Тест: «Сказал бы senior-инженер, что это переусложнено?» Если да — упрощай.

### 3. Хирургические изменения

**Трогай только то, что нужно. Убирай только свой мусор.**

- Не «улучшай» соседний код без запроса.
- Следуй существующему стилю.
- Каждая изменённая строка должна прослеживаться к запросу пользователя.

### 4. Целевое исполнение

**Определи критерии успеха. Итерируй до верификации.**

Трансформируй задачи в проверяемые цели:
- «Добавь форму» → «Форма валидирует поля X, Y, Z и отправляет на endpoint /api/...»
- «Почини баг» → «Напиши тест, воспроизводящий баг, затем исправь»

### 5. Документы — часть фичи. ВСЕГДА обновлять зависимые

**Любая новая фича / изменение поведения / новый процесс — это не только код. Это ещё и обновление всех документов, которые от него зависят. Без этого фича считается незаконченной.**

Правила:

- **Перед началом работы:** найди все документы, которые ссылаются на затрагиваемую область (ТЗ, JTBD, user stories, ARCH, OPS, SOS, FINANCE, LEGAL, UX, LOYALTY, BUSINESS_MODEL). Список зависимостей — в [`docs/TZ/README.md`](docs/TZ/README.md).
- **Если фича вводит новый объект / роль / поле / SLA / API** — обнови:
  - Соответствующий раздел `docs/TZ/`.
  - `docs/USER_STORIES.md` (новая история или обновление acceptance).
  - `docs/JTBD.md`, если меняется работа, которую делает клиент или команда.
  - Глоссарий в `docs/TZ/README.md` — если появился новый термин.
- **Если фича касается денег / маржи** — обнови `docs/BUSINESS_MODEL.md` и `docs/BUSINESS_MODEL/UNIT_ECONOMICS.md`.
- **Если фича касается ПДн, оферты, согласий** — обнови `docs/LEGAL/`.
- **Если фича касается SOS, операций или гидов** — обнови `docs/SOS/` и/или `docs/OPS/`.
- **Если фича касается архитектуры** — обнови `docs/ARCH/`.
- **Если фича — клиентская UX** — обнови `docs/UX/`.
- **Если изменился каркас репозитория** — обнови `CLAUDE.md`, `CONTRIBUTING.md`, `README.md`.
- **Каркас-индекс `docs/TZ/README.md` и `docs/TZ/MVP_PHASE3.md`** — обновляй при добавлении/закрытии любого блока: статусы эпиков, ссылки на новые файлы, `Acceptance criteria` чек-боксы.
- **Удалил функциональность — удалили ссылки.** Мёртвых ссылок в документах не оставляем.
- **Документ и код — в одном PR.** Если код мерджится без обновления зависимых доков, PR не принимается.

Если непонятно, какие документы зависят от изменения — сначала спроси / составь список и сверься, потом меняй код.

### 6. Проактивные рекомендации с объяснением «почему»

**Не молчи о том, что знаешь. Везде, где работаешь — issue, документ, фича, код-ревью — давай рекомендации, опираясь на опыт travel-индустрии и специфику Индии. И всегда объясняй, почему это важно.**

Контекст: founders сознательно хотят использовать насмотренность ассистента — по тревелу, по операциям в Индии, по похожим продуктам — как часть рабочего процесса. Молчаливое исполнение задачи без рекомендаций здесь считается недоработкой.

Правила:

- **Если видишь риск или антипаттерн** — назови его. Пример: «отзыв за скидку — нарушает 152-ФЗ и убивает доверие к UGC, потому что …».
- **Если видишь возможность, которую не запросили** — предложи отдельным пунктом, не молча добавляй в код. Решение принимают founders.
- **Если в индустрии есть устойчивая практика** — упомяни и объясни, почему она применима или неприменима у нас. Пример: «обычно в DMC делают X, но у нас фаза 3 / другой клиент / другая маржа, поэтому …».
- **Если в Индии есть локальная специфика** (TDS, поведение поставщиков, сезонность, регуляторика, культурные нюансы) — упомяни её даже если не спрашивали.
- **Каждая рекомендация — с обоснованием.** Не «давайте сделаем X», а «давайте сделаем X, потому что Y → Z». Без «почему» рекомендация считается неполной.
- **Маркируй рекомендации явно.** В документах и issues — отдельным блоком `> **Рекомендация:** …` или `### Рекомендация (опыт)`. В коде — комментарием с пометкой `// рекомендация:`.
- **Альтернативы — тоже рекомендация.** Если у решения есть 2–3 варианта с разным риском/маржей/сложностью — перечисли их с трейдоффами, не выбирай молча.
- **Не превращай в шум.** Рекомендация — там, где она реально может изменить решение. Тривиальные «считаю что лучше так» без последствий — не нужны.

Тест: «Senior-консультант по travel в Индии увидел бы здесь риск или возможность, которую я скрыл?» Если да — рекомендация обязательна.

### 7. Серверные действия — через issue для Вики (devops AI)

**Любое действие, которое требует доступа к серверу / окружению / стороннему сервису — НЕ выполняется ассистентом самостоятельно. Создаётся issue с подробными инструкциями для Вики (devops AI).**

Контекст: разделение ответственности. Ассистент пишет код и документацию, Вика выполняет операции в инфраструктуре. У ассистента нет доступа к prod/staging серверам, к секретам, к billing-аккаунтам провайдеров — и не должно быть.

#### Когда создаётся issue для Вики

- Установка / обновление пакетов на сервере (`apt`, `pnpm install` локально на dev-машине, `docker pull`)
- Запуск/перезапуск контейнеров на staging/prod
- Создание / обновление DNS-записей
- Создание ресурсов в облачных провайдерах (S3 buckets, fly.io apps, базы данных)
- Заведение секретов в Vault / KMS
- Настройка GitHub repository settings (защита веток, secrets, webhooks, labels через UI)
- Настройка платёжных шлюзов (ЮKassa, Tinkoff и т.п.)
- Настройка SMS/push-провайдеров (Twilio, FCM, APNs)
- Любая верификация локального окружения (`pnpm install`, `docker compose up`, проверка healthchecks)
- Любая ротация ключей / паролей

#### Формат issue для Вики

Title: `[devops:vika] <короткое описание задачи>`

Body — структурированный, с командами:

```
> Исполнитель: Вика (devops AI).
> Контекст: <откуда задача, ссылки на родительские issues>.

## Цель

<Один-два предложения — что должно быть на выходе>

## Окружение

- Где выполнить (local dev / staging / prod)
- Требуемые credentials (что должно быть доступно Вике)
- Pre-requisites (что должно быть сделано до)

## Команды (по шагам)

1. `command 1`
2. `command 2`
3. ...

Для каждого шага — **ожидаемый результат** + **что делать если упало**.

## Verification

Чек-лист как убедиться что всё работает.

## Что вернуть в issue

- Логи выполнения
- Скриншоты/выводы команд
- Список файлов / ресурсов, которые изменились
- При ошибках — diagnostic-information

## Acceptance criteria

- [ ] Команды выполнены
- [ ] Verification пройден
- [ ] Отчёт прикреплён
```

#### Что Вика **не делает**

**Главное правило:** Вика — devops AI, не fullstack-разработчик. Продуктовый код пишет ассистент / dev-команда. Граница «продуктовый код vs инфра» проводится так:

| ❌ НЕ делает Вика (это код ассистента) | ✅ Делает Вика (инфра) |
|---|---|
| Prisma модели (`User`, `Client`, `Trip`...) | `prisma migrate dev/deploy` |
| Миграции `migration.sql` (если они меняют бизнес-схему) | Применение миграций к staging/prod |
| NestJS-модули (`AuthModule`, `ClientsModule`...) | `pnpm install`, проверка деп-конфликтов |
| Services / Controllers / Listeners / Workers | Запуск `pnpm dev:api`, `make up`, smoke-tests |
| DTO, validation, бизнес-логика | Логи / output этих команд |
| React-компоненты, hooks, pages | Деплой в fly.io / Hetzner |
| Тесты (Jest, Playwright) — **код тестов** | CI workflow конфиги (если бизнес-зависимы) |
| Архитектурные решения (карта сервисов, event-каталог) | Vault / KMS, secrets-rotation |
| Любой `.ts`/`.tsx`/`.prisma`-файл с бизнес-логикой | DNS, CDN, S3 buckets |
| Issue body / governance docs | GitHub repo settings: labels, branch protection, webhooks |

**Если Вика получила задачу на продуктовый код** (например «реализуй #138 модели Client»):
1. Закрывает issue без действий, оставляя комментарий: «Это задача ассистента (продуктовый код). Перенаправлено».
2. Ассистент берёт issue в работу.
3. Если граница неясна (например, миграция, которая одновременно код и инфра) — Вика спрашивает founders в комментарии issue, не делает молча.

**Прочие ограничения:**

- Не утверждает архитектурные решения
- Не делает destructive операции (удаление БД, force-push в main, удаление S3-buckets) без явного подтверждения founders в issue
- Не публикует pre-release версии без аппрува

> **Почему так:** разделение даёт чёткий audit trail и фокус каждого AI. Ассистент видит full-stack контекст (backend → frontend → e2e тесты), Вика видит инфру (deploy, secrets, performance). Смешение → теряется концепция «два специализированных AI» в пользу «два generic AI», и backend без UX-контекста рискует не совпадать с frontend (типы, hooks, контракт API).

#### Связь с другими принципами

- Принцип 5 (документы — часть фичи): issue для Вики — это часть фичи. Без её закрытия фича не считается готовой.
- Принцип 6 (рекомендации): в issue для Вики тоже даём рекомендации — например, «если `pnpm install` падает с peer-deps конфликтом, скорее всего нужно проверить версию `react` в `apps/web/package.json`».

### 8. Иерархия GitHub Issues — epic / issue / sub-issue

**Все задачи в трекере живут в трёхуровневой иерархии: эпик → issue → sub-issue. Любой новый issue без явного родителя — нарушение, кроме самих эпиков.**

Контекст: при ~200+ открытых задач плоский список превращается в шум. Иерархия (через нативную GitHub sub-issues feature) даёт визуальный группинг — founders и dev'ы видят, какому эпику принадлежит каждый кейс, какой эпик пора закрывать, где скопилось техдолга.

#### Уровни

| Уровень | Что | Пример |
|---|---|---|
| **Epic** | Стратегический блок продукта на квартал/год. Закрывается, когда закрыты все вложенные. Title начинается с `[EPIC N]`. | `[EPIC 6] Архитектура — modular monolith → микросервисы` |
| **Issue** | Конкретная фича или задача в рамках эпика. Может закрываться одним PR или серией связанных PR'ов. Title — Conventional-Commit-style, опционально с префиксом `[N.M]` или `[M5.A.1]`. | `[M5.A.1] chore(repo): scaffold pnpm workspaces` |
| **Sub-issue** | Атомарный шаг внутри issue, если issue декомпозируется. Реализуется одним PR. | `chore(api): add health endpoint route` (под issue NestJS scaffold) |

> **Замечание:** в фазе 3 большинство наших backlog-задач — на уровне **issue**, не sub-issue. Sub-issue появляются, когда отдельная задача делается в 3+ PR'ах (например, «внедрить аутентификацию» → sub-issues: User модель, login API, refresh token, 2FA enrollment).

#### Правила

1. **Каждый issue — потомок какого-то эпика.** Привязывается через GitHub native sub-issue (предпочтительно) или через body эпика с чек-листом ссылок на issues.
2. **Каждый эпик имеет:**
   - Title `[EPIC N] <короткое название>`
   - Цель / scope / acceptance / связанные документы
   - Чек-лист sub-issues (для UI и быстрого обзора)
3. **Появилась идея — открываем issue под существующим эпиком.** Если идея масштаба нового эпика (новый продуктовый блок) — открываем новый эпик, потом issues.
4. **Issue может относиться тематически к нескольким эпикам.** Primary parent — наиболее подходящий по основной зоне. Дополнительные связи — через `Related to #X` в body.
5. **Не создавать «голых» issues.** Перед `Create issue` спрашиваем: «к какому эпику?». Если не знаешь — спрашиваешь founders.
6. **При reorganization** (перераспределении в другой эпик) — обновляем родителя через sub-issue API + body старого/нового эпика, чтобы не оставлять мёртвых ссылок.

#### Когда создавать **новый** эпик

Триггеры (хотя бы один из):
- Новый продуктовый блок, который раньше не был в скоупе (например, добавление мобильного приложения для founders).
- Новая регуляторная зона (например, GDPR при выходе на ЕС-клиентов).
- Объём 10+ issues с чёткой границей.
- Founders явно решили выделить как самостоятельный milestone.

Не каждая идея = новый эпик. «Добавить export PDF в дневник» — это **issue** под `EPIC 7 (UX и фичи)`, а не «EPIC 11 PDF Export».

#### Связь с принципами

- Принцип 5 (документы — часть фичи): новые эпики и крупные issues добавляются в [`docs/TZ/README.md`](docs/TZ/README.md) и [`docs/TZ/MVP_PHASE3.md`](docs/TZ/MVP_PHASE3.md).
- Принцип 6 (рекомендации): при создании эпика — обязательно проговариваем «почему именно эпик, а не issue под существующим».
- Принцип 7 (Вика): создание / переименование эпиков в GitHub UI — это операция в инфраструктуре. Если требуется массовая перепривязка sub-issues — отдельный issue для Вики; если 1–5 точечных изменений — ассистент делает через MCP.

## О проекте

**IndiaHorizone** — нишевой travel-tech сервис для русскоязычных клиентов.

### Позиционирование

> Не турагентство. Не просто travel. А tech-enabled India concierge для русскоязычного клиента.

- **Продукт** = Индия без хаоса
- **Технология** = ускоряет продажи и сервис
- **Локальная сеть** = даёт реальную опору на месте
- **Сопровождение** = создаёт доверие и высокую ценность

### Модель развития

1. Кастомные туры и high-touch сервис
2. Автоматизация и повторные продажи
3. Масштаб через групповые форматы и тематические продукты

## Git-воркфлоу

Ветки:
```
feature/описание-номер-issue
fix/описание-номер-issue
docs/описание-номер-issue
```

Коммиты — Conventional Commits:
```
feat(scope): описание
fix(scope): описание
docs(scope): описание
refactor(scope): описание
test(scope): описание
chore(scope): описание
```

Примеры:

### TypeScript

- Именованные экспорты

- Функциональные компоненты с `FC<Props>`
- Интерфейс пропсов рядом с компонентом

- Оффлайн-first: все критичные данные кешируются локально
- Учитывать нестабильный интернет в Индии (retry, graceful degradation)
- Поддержка iOS и Android

## Принципы продукта

1. **Оффлайн-first** — клиент в Индии не должен зависеть от стабильного интернета
2. **Mobile-first** — основные сценарии проектируются для телефона
3. **Русскоязычный UX** — весь интерфейс на русском, без английских вставок
4. **Доверие через прозрачность** — клиент всегда знает, что происходит и что будет дальше
5. **Минимум шагов** — любое действие (SOS, фидбэк, запрос доп. услуги) — максимум 2 тапа

## 📋 Workflow для Claude Code

### Перед началом задачи
1. Прочитай issue полностью — заголовок, описание, criteria of done
2. Проверь связанные issues (Part of #N)
3. Посмотри существующий код — не изобретай велосипед
4. Создай ветку по соглашению: `feature/m5-{epic}-{num}-{slug}-{issue}`

### В процессе
- Коммиты по Conventional Commits: `feat(scope): описание`
- Один коммит = одно логическое изменение
- Не трогай несвязанный код

### Перед PR
- `pnpm test` — все тесты зелёные
- `pnpm lint` — нет ошибок
- `pnpm type-check` — нет TypeScript ошибок
- Описание PR по шаблону (выбирается автоматически)

### После merge
- Ветка удаляется
- Issue закрывается через `Closes #N` в PR
