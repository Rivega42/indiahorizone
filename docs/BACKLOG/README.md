# Backlog — атомарные задачи по разработке

> **Owner:** Roman.

## Что это

Backlog — это **готовые к копированию в GitHub Issues** задачи, атомизированные так, чтобы:

- одна задача = ≤ 8 часов работы (один день максимум, чаще 2–4 часа)
- одна задача = один PR
- задача имеет **ясное Acceptance criteria**, которое можно проверить
- задача имеет ссылки на архитектуру/доки, которые её обосновывают

## Зачем держать в репо, а не сразу в GitHub Issues

1. **Backlog ходит вместе с архитектурой.** Поменялась карта сервисов в `docs/ARCH/MICROSERVICES.md` → правим backlog в том же PR, не теряем согласованность.
2. **Меньше шума в трекере.** Создавать сразу 100+ issues = заваливать GitHub. Сначала формируем pipeline здесь, потом массово выносим.
3. **Легко переписывать.** Реструктурировать markdown в репо легче, чем массово редактировать GitHub Issues.

## Соглашения

### Формат ID

`IH-M<milestone>-<slice>-<n>` — например `IH-M5-A-001` (M5, slice A, задача 1).

### Поля

```
### <ID> — <conv-commit-style title>
- Slice: <название slice>
- Type: feat / fix / chore / docs / test
- Estimate: Nh
- Owner-role: dev / mobile / frontend / backend / devops / qa / pm
- Deps: [<ID>, <ID>] | —
- Acceptance:
  - [ ] критерий 1
  - [ ] критерий 2
- Files (примерно): paths
- Labels: area:* slice:* priority:* type:*
- Notes (опционально): пояснения, ссылки на ARCH/UX/USER_STORIES
```

### Slices

Slice = вертикальный срез функциональности. Один slice доводится до состояния «можно показать клиенту» прежде чем переходить к следующему.

### Приоритеты

- `p0` — без этого slice не закрывается
- `p1` — нужно в slice, но можно отложить на 1 неделю
- `p2` — желательно, опционально

### Перенос в GitHub

Когда backlog утверждён — массовое создание через `mcp__github__` или скрипт, **по согласованию с founders**. До этого — issues остаются здесь.
