## Контекст

- **152-ФЗ:** геолокация, привязанная к идентифицированному человеку, = специальная категория ПДн.
- **GDPR-аналогия:** location data — sensitive, требует extra-обоснования.
- **Apple/Google guidelines:** в мобильных приложениях — отдельный prompt, отдельная mission justification.

## Зачем нам гео

| Цель | Почему обоснованно |
|---|---|
| SOS-реагирование | Без точной геолокации концьерж не может направить помощь в Индии. Это **критичная** функция. |
| Подтверждение прибытия гида/клиента в trip-хаб | Контроль качества операций |
| Геофенсинг для пушей («вы рядом с …») | UX-усиление (некритично) |
| Анализ маршрутов post-trip | Только обезличенно |

> **Рекомендация:** разделить «обязательную для SOS» гео и «опциональную для UX». **Почему важно:** если клиент откажется от гео целиком, мы не можем гарантировать SOS — это нужно проговорить явно. И при этом «обязательная гео» должна быть **только для активной поездки** (не 24/7 круглый год).

## Принципы

1. **Granular**: SOS-гео и UX-гео — отдельно.
2. **Только в активной поездке**: «Live tracking» включается за 24 часа до старта и выключается через 24 часа после возвращения.
3. **Минимум данных**: координаты округляются до точности, нужной для задачи. Для SOS — точные. Для геофенсинга — округлённо.
4. **Прозрачность**: в Trip Dashboard клиент всегда видит, передаётся ли его гео сейчас или нет (индикатор).
5. **Off в любой момент**: одна кнопка в Trip Dashboard.

## Что включено по умолчанию (рекомендация Оператора)

При первом входе в Trip Dashboard клиент видит **рекомендованную** комбинацию:

- ✅ Уровень A (SOS) — по умолчанию ON
- ✅ Уровень B (Live-tracking в активной поездке) — по умолчанию ON
- ⬜ Уровень C (геофенсинг) — по умолчанию OFF, опционально
- ⬜ Уровень D (аналитика) — по умолчанию OFF, опционально

Клиент может изменить любой уровень в любой момент.

> **Рекомендация:** не делать ON по умолчанию для C и D. **Почему важно:** GDPR-style принцип «privacy by default». Если клиент сам не включил — не включаем. Это поднимает доверие и снижает регуляторный риск.

## Технические правила обработки

| Правило | Зачем |
|---|---|
| Передача через TLS 1.3+ | Защита от перехвата |
| Хранение в `geo-svc` отдельно от `clients-svc` | Минимизация blast radius при утечке |
| Доступ — только sea-shift concierge на смене + audit | Принцип минимума привилегий |
| Ретенция автоматическая по таймеру | Не полагаемся на ручное удаление |
| При выходе клиента из Индии (по геолокации **или** по графику поездки) — Уровень B автоматически выключается | Не следим вне поездки |

## Прозрачность

В Trip Dashboard всегда виден индикатор:

```
🟢 Гео передаётся (Уровень B активен)
🔴 Гео не передаётся
🆘 Гео передано в SOS — <время>
```

В CRM (со стороны Оператора) — те же состояния, видимо концьержу и гиду на смене.

## Что мы НЕ делаем

- НЕ продаём данные третьим лицам, никогда
- НЕ используем гео для рекламного таргетинга
- НЕ агрегируем гео в коммерческие отчёты, доступные за пределами IH
- НЕ передаём гео клиента другим клиентам или гидам, не связанным с его поездкой
- НЕ запрашиваем гео в фоне приложения, когда поездка не активна

## Несовершеннолетние

- Согласие даёт законный представитель
- Live-tracking — рекомендуется ON по умолчанию (защита ребёнка > privacy)
- При исполнении 18 лет — согласие **переоформляется** (если клиент не возражает)
