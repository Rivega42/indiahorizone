# Чек-листы гида — ежедневные SOP

> **Статус:** Draft v0.1. Часть EPIC 4 [#31](https://github.com/Rivega42/indiahorizone/issues/31).
> **Owner:** Shivam.
> Связано: [`GUIDE_ONBOARDING.md`](./GUIDE_ONBOARDING.md), [`STRUCTURE.md`](./STRUCTURE.md), [`CLIENT_ONBOARDING.md`](./CLIENT_ONBOARDING.md).

## Зачем

Превратить «хорошего гида» в «предсказуемого гида». Без формализованного SOP качество поездки целиком зависит от характера гида. С SOP — от характера и системы, которая страхует на минимуме.

> **Рекомендация (опыт):** чек-лист — не бюрократия, а **подушка безопасности гида**. Если гид забыл что-то критичное — это не «он плохой», это «процесс плохой». При обсуждении с гидами всегда подавать SOP так: «это твой щит, если что-то пошло не так — у тебя в руках доказательство, что ты сделал всё, что должен». **Почему важно:** гиды-индийцы культурно более чувствительны к «контролю», чем к «защите» — переформулировка снимает сопротивление к чек-листам.

## §1. Pre-trip (за 24–48 часов до встречи)

- [ ] **Подтверждение от concierge:** клиент летит, ETA в аэропорту
- [ ] Изучен профиль клиента в CRM:
  - [ ] ФИО, фото
  - [ ] Аллергии и диета (вегетарианство, без говядины, без молочки и т.п.)
  - [ ] Темп (медленный / средний / активный)
  - [ ] Дети, маломобильные участники
  - [ ] Особые пожелания (предпочтения по фотографии, тишина, разговорчивость)
  - [ ] Медицинские заметки (хронические, аптечка)
- [ ] Прочитана программа на ВСЮ поездку, не только первый день
- [ ] **Ваучеры на отели на руках** (PDF / печать), сверены имя-даты
- [ ] **Бронирования на трансферы и активности** подтверждены ещё раз (отель, водитель)
- [ ] Запас наличных (INR) на день — ≥ ₹X (Shivam утверждает по сезону)
- [ ] Заряжен телефон, есть power bank, активная локальная SIM с интернетом
- [ ] **Бумажная SOS-карточка** при себе (см. [`docs/SOS/FALLBACK.md`](../SOS/FALLBACK.md))
- [ ] Аптечка проверена (см. §8 ниже)
- [ ] **Связь с предыдущим гидом** (если поездка multi-region и его сменяют) — устный пересказ важных нюансов

> **Рекомендация:** baton-pass между гидами всегда устный + письменный (заметка в CRM). **Почему:** теряется 30% контекста при только-письменной передаче. Голосом гид расскажет «клиент очень болезненно реагирует на упоминание X» — то, что в текст не попадёт.

## §2. Встреча в аэропорту

### За 90 минут до прилёта

- [ ] Проверить статус рейса (на табло онлайн / приложение)
- [ ] Если задержка > 30 минут — уведомить concierge → пуш клиенту
- [ ] Прибыть в аэропорт за 60 минут до посадки

### При выходе клиента

- [ ] **Табличка с именем клиента** (русский + латиница) — стоять в зоне выхода
- [ ] Приветствие на русском, представиться по имени
- [ ] **Селфи с клиентом** + отправка в чат «IH-trip-<id>» в Telegram (отметка прибытия)
- [ ] Уточнить: SIM-карта (нужна ли), вода, туалет
- [ ] Объяснить план первых 2 часов (трансфер до отеля, чек-ин)

### В трансфере

- [ ] Подтвердить водителю маршрут и адрес
- [ ] Передать клиенту бутылку воды
- [ ] Не перегружать клиента информацией — у него jet lag

> **Рекомендация:** молчание в первые 30 минут после прилёта — это тоже сервис. **Почему:** русский клиент после 6+ часов перелёта в Индию обычно перегружен сенсорно (звуки, запахи, толпа). Гид, который сразу включает «культурный гид-режим», даёт обратный эффект. Отдохнуть, потом узнавать.

### По прибытии в отель

- [ ] Помочь с чек-ином (язык + ваучер + паспорт)
- [ ] Проверить номер: чистота, кондиционер, Wi-Fi работает, душ работает, замок работает
- [ ] Если что-то не так — **сразу** к ресепшен, не ждать клиента
- [ ] Объяснить «здесь рядом …» (вода, аптека, такси, безопасные/небезопасные зоны)
- [ ] Согласовать время встречи завтра
- [ ] Оставить контакт (WhatsApp + emergency)

## §3. Утро каждого дня

### До встречи с клиентом

- [ ] Программа дня сверена ещё раз (нет ли изменений от concierge)
- [ ] Бронирования на сегодня подтверждены (обзвон / WhatsApp партнёров)
- [ ] Погода + чрезвычайные новости региона проверены
- [ ] Аптечка пополнена при необходимости
- [ ] Наличные на день — есть
- [ ] Зарядка устройств — на 100%

### При встрече

- [ ] Спросить, как спалось / как самочувствие
- [ ] Уточнить аппетит, готов ли к завтраку
- [ ] **Открыто проговорить план дня** (даже если в Trip Dashboard)
- [ ] Спросить «что-то особенное хочешь сегодня?»

> **Рекомендация:** короткий утренний overview даже при наличии Trip Dashboard. **Почему:** клиент в путешествии часто не открывает приложения, и устное проговаривание плана — это не дублирование, а сигнал «я держу твой день в голове».

## §4. В течение дня

### Постоянные правила

- [ ] **Никогда не оставлять клиента одного** в публичных зонах с риском (рынок, вокзал, толпа на празднике) без явного «я отойду на 10 минут, ждите тут с бутылкой воды»
- [ ] Каждые 90 минут — предложение воды
- [ ] Каждые 2 часа — предложение туалета
- [ ] Каждый час физической активности — предложение паузы
- [ ] Фото в режиме «гид-фотограф»: 5–10 хороших фото в день (не «100 на коленке»)
- [ ] Запись 30–60-секундного фрагмента для альбома клиента (по согласию — см. [`docs/LEGAL/CONSENTS/PHOTO_VIDEO.md`](../LEGAL/CONSENTS/PHOTO_VIDEO.md))

### Финансовая прозрачность

- [ ] При любой оплате клиент видит цену **до** транзакции
- [ ] Чеки/билеты сохраняются (фото или физически) — для отчётности
- [ ] **Никогда** не предлагать «заехать в магазин знакомого» без отдельной просьбы клиента
- [ ] Если клиент хочет купить что-то — рассказать о реальном диапазоне цен **до** того, как отвести в магазин

### Безопасность

- [ ] Геолокация передаётся в Trip Dashboard (с согласия [`docs/LEGAL/CONSENTS/GEO.md`](../LEGAL/CONSENTS/GEO.md))
- [ ] При смене водителя — фото водителя и номера машины в чат «IH-trip-<id>»
- [ ] При входе на территорию с риском (ночной рынок, фестиваль) — кратко проинструктировать клиента

### Если клиент заскучал / устал / расстроен

- [ ] Не игнорировать. Прямо спросить «всё ок? хочешь паузу / поменять план?»
- [ ] Если изменение программы возможно — предложить варианты
- [ ] Если нужно решение от менеджера — связаться с concierge

## §5. Вечерний debrief (19:00–21:00 локально)

### С клиентом

- [ ] Дойти до отеля, проводить, попрощаться до утра
- [ ] Тёплое прощание, не сухое «до завтра»
- [ ] Напомнить «вечером придёт push с просьбой записать кружок — он бесценен для нас»

### Самостоятельно

- [ ] **Загрузить фото дня** в альбом клиента (через приложение / WhatsApp концьержу для архива)
- [ ] **Загрузить кружок-фрагмент дня** (если согласие есть)
- [ ] **Заполнить debrief** в чате «IH-trip-<id>»:
  - Эмоциональное состояние клиента (😊 / 😐 / 😔)
  - Прошло по плану / отклонения
  - Жалобы / пожелания
  - Расходы дня (с чеками)
  - Прогноз на завтра (риски, что подкорректировать)
- [ ] **Чеки** сфотографированы и привязаны к расходам дня (`см. учёт расходов, TBD`)
- [ ] При негативе клиента — пуш концьержу с пометкой «нужна реакция»

### При негативном фидбэке клиента

- [ ] Не молчать — связаться с концьержем смены сразу, не дожидаться утра
- [ ] Предложить варианты компенсации в рамках своих полномочий (ужин за счёт IH, smoothie, цветы)
- [ ] Эскалировать выше при необходимости (см. [`ESCALATION.md`](./ESCALATION.md))

> **Рекомендация:** в фазе 3 первый час после негативного фидбэка — окно, где можно вернуть клиента. **Почему:** через 12 часов он уже расскажет всё мужу/жене/друзьям, и обратной дороги нет. Скорость > извинений.

## §6. Конец поездки

### Последний день

- [ ] Все чеки и расходы переданы
- [ ] Все фото и кружки загружены
- [ ] Прощальный фрагмент-кружок гида клиенту (короткий «спасибо за поездку»)
- [ ] Сопровождение в аэропорт
- [ ] Отправка пуш «клиент в аэропорту» в чат
- [ ] Подтверждение чек-ина клиента на рейс
- [ ] Отправка прощального push (через концьержа) после посадки

### После поездки

- [ ] Внутренний отчёт по поездке в CRM:
  - Что прошло хорошо
  - Что можно улучшить (поставщики, маршрут, активности)
  - Личная заметка о клиенте на случай повторной поездки
- [ ] Финансовая сверка с finance — закрыть расходы
- [ ] При негативных моментах — личный звонок/встреча с Shivam

## §7. Внештатные ситуации

### Триггеры — какие playbook'и применять

| Ситуация | Playbook |
|---|---|
| Клиент почувствовал себя плохо | [`docs/SOS/PLAYBOOKS/MEDICAL.md`](../SOS/PLAYBOOKS/MEDICAL.md) |
| Авария / поломка трансфера | [`docs/SOS/PLAYBOOKS/TRANSPORT.md`](../SOS/PLAYBOOKS/TRANSPORT.md) |
| Полиция, конфликт | [`docs/SOS/PLAYBOOKS/POLICE.md`](../SOS/PLAYBOOKS/POLICE.md) |
| Утрата документов | [`docs/SOS/PLAYBOOKS/DOCUMENTS.md`](../SOS/PLAYBOOKS/DOCUMENTS.md) |

### Универсальные правила в любой внештатной ситуации

1. **Сначала безопасность клиента** — не геройствовать
2. **SOS — за 60 секунд** (Trip Dashboard либо звонок концьержу)
3. **Не врать клиенту** — даже если ситуация плохая, говорить правду в спокойной форме
4. **Не платить «штрафы» полиции / водителям** без согласования с концьержем

> **Рекомендация:** «не платить взятку» — критично документировать. **Почему:** в Индии чрезвычайно распространены «требования» от police / traffic / customs, которые на самом деле необязательны и часто незаконны для иностранцев. Платить вне playbook'а = ставить клиента и компанию в правовой риск. Лучше потерять час и привлечь концьержа/Shivam.

## §8. Аптечка гида (стандарт)

| Категория | Минимум |
|---|---|
| От расстройства ЖКТ | Смекта / Imodium, регидрон |
| Жаропонижающее | Парацетамол |
| Обезболивающее | Ибупрофен |
| Антигистамин | Цетиризин |
| Антисептик | Перекись водорода / Бетадин, бинт, пластыри |
| Защита от солнца | SPF50, after-sun |
| Москиты | Repellent + крем после укусов |
| Пероральные регидраторы | ORS пакеты |
| Дополнительно по сезону | Anti-malaria тестер (только в эндемичных регионах, по назначению) |

> **Рекомендация:** в аптечку **не класть** ничего рецептурного для клиента. **Почему:** даже из лучших побуждений — это юридический риск, и взаимодействие с препаратами клиента непредсказуемо. Аптечка гида = first-aid, не лечение.

---

## §9. Что гид НЕ делает

- НЕ продаёт клиенту дополнительные услуги без концьержа
- НЕ берёт чаевые в обход IH (если клиент даёт лично — благодарит и фиксирует в отчёте)
- НЕ сводит клиента с поставщиками IH «напрямую» в обход IH
- НЕ публикует фото/видео клиента в соцсетях без явного согласия (см. [`docs/LEGAL/CONSENTS/PHOTO_VIDEO.md`](../LEGAL/CONSENTS/PHOTO_VIDEO.md))
- НЕ делится контактами клиентов между гидами
- НЕ обсуждает других клиентов с текущим клиентом
