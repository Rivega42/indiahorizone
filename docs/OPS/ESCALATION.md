## Принцип

Эскалация — это **запрос помощи**, не «жалоба». Эскалирующий не теряет ответственность, но делит её с тем, кто получил эскалацию.

## Кто кому эскалирует

```
concierge → sales / manager (зависит от темы)
sales / manager → Roman или Shivam (по зоне)
Roman или Shivam → совместное решение founders
```

Параллельно (по событию):
- SOS любого типа → автоматически Shivam (всегда), Roman (если в зоне действий)
- Финансовые конфликты — finance + Roman
- Юридические — внешний юрист + Roman

## Триггеры эскалации

| Триггер | Куда эскалировать | За сколько минут / часов |
|---|---|---|
| SOS breach SLA (60 сек / 3 мин) | резерв + founders | сразу (auto) |
| Незакрытое срабатывание SOS > 30 мин | founders | сразу |
| Жалоба клиента публичная (отзовик) | sales + Roman | ≤ 1 час |
| Конфликт с поставщиком (отель, гид) | Shivam | ≤ 4 часа |
| Финансовое расхождение > ₹50 000 | finance + Roman | ≤ 24 часа |
| Юр. вопрос от клиента (претензия, повестка) | Roman + юрист | ≤ 4 часа |
| Подозрение на AML-флаг | Roman + finance | сразу |
| Гид не вышел на смену без предупреждения | Shivam + резервный гид | сразу |
| Запрос Роскомнадзора | Roman + юрист | ≤ 24 часа |
| Запрос ФНС / банка | finance + Roman | ≤ 24 часа |
| Проблема со здоровьем сотрудника, который на смене | другой founder | сразу |

## Каналы эскалации (приоритизированы)

1. **Звонок** (Twilio автозвонок или личный) — для критичных
2. **Telegram личное сообщение** — для срочных
3. **Telegram группа «IH-ops»** — для большинства
4. **Email** — для документарных запросов

## Pager-rotation для founders

| День недели | Roman | Shivam |
|---|---|---|
| Пн–Пт | primary on-call | secondary on-call |
| Сб–Вс | secondary on-call | primary on-call |

«primary on-call» = его звонит auto-pager в первую очередь (через Twilio при escalation event).
«secondary on-call» = звонят, если primary не ответил за 5 минут.

В отпуске founder уведомляет за 7 дней + назначает альтернативного primary (или второй founder покрывает).

## Регламент после эскалации

Получивший эскалацию:

1. Подтверждает приём сообщением «принял, разбираюсь, имя X»
2. Действует по своей зоне ответственности
3. Возвращается с результатом или планом в течение указанного SLA для эскалации
4. Не «откатывает» обратно без причины (новый owner — это новый owner)

## Что НЕ эскалировать

- Рутинные вопросы клиента, на которые есть готовый ответ → решает concierge
- Стандартные операции в рамках playbook (уже описанные сценарии) → дежурный сам
- «Просто сложно» — не эскалация, а тренинг и менторинг
