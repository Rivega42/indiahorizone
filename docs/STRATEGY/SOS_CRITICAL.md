## Почему SOS — это не просто «кнопка»

SOS — это **обещание**, которое уже дано клиенту. Оно вписано в оферту. Оно продаёт.

Но сейчас SOS держится на одном человеке: Шивам отвечает 24/7 личным телефоном.

Это три проблемы одновременно:

**1. Операционный риск**
Шивам заболел / уехал / не взял трубку → обещание нарушено → репутационный ущерб → возможный судебный иск (нарушение оферты).

**2. Блокер масштабирования**
При 20 поездках одновременно — 1 человек физически не может обработать несколько SOS параллельно.

**3. Блокер для платформы**
Другой оператор не купит SOS-as-a-Service, если у нас самих система не работает без основателя.

## Как должно работать

### Принцип: система, а не герой

SOS должен работать как **дежурная смена**, а не как «звони Шиваму».

```
Клиент нажал SOS
        ↓
Автоматический алерт → дежурному concierge (не Шиваму)
        ↓ (если нет ответа 60 сек)
Эскалация → второй дежурный
        ↓ (если нет ответа 3 мин)
Эскалация → Шивам (только как последний уровень, не первый)
        ↓
Параллельно: Roman получает push (информация, не действие)
```

## Функционал SOS-модуля

### Фаза 3 (MVP минимум)

**Для клиента:**
- Кнопка SOS в Trip Dashboard (большая, красная, одним нажатием)
- При нажатии: геолокация + последняя активность маршрута → автоматически прикрепляются к тикету
- Статус: «Принято», «В работе», «Решено» — клиент видит в реальном времени
- Fallback: если нет интернета → SMS на выделенный номер (Twilio)
- Карточка клиента в SOS-тикете: паспорт, страховая, экстренный контакт, медицинские особенности, аллергии

**Для дежурного concierge:**
- Мобильное приложение (или PWA) с push + звонок при SOS
- Скрипт первых вопросов прямо в интерфейсе (не в голове)
- Быстрые действия: «Связать со страховой», «Вызвать такси», «Связаться с отелем», «Полиция»
- Шаблоны сообщений клиенту на русском
- Таймер SLA — видит, сколько секунд до breach

**Для Шивама/Roman (операционный центр):**
- Дашборд: все активные инциденты, статус, время реакции
- История инцидентов с постинцидентным разбором
- Алерт при SLA breach

### Фаза 4 (SOS-as-a-Service)

**Дополнительно:**
- Пул дежурных concierge — не зависит от конкретного человека
- Распределение по часовым поясам: дежурный в IST для ночи по РФ
- Запись разговоров (с согласия) — для постинцидентного разбора и обучения
- AI-ассистент дежурного: подсказывает следующий шаг по типу инцидента
- Мультиязычность: клиент пишет по-русски, шаблоны переведены на английский/хинди для местных партнёров

## Смены 24/7 — минимальная структура фазы 3

| Смена | Время (IST) | Кто дежурит | Резерв |
|---|---|---|---|
| Дневная | 06:00–14:00 | Concierge A | Shivam |
| Вечерняя | 14:00–22:00 | Concierge B | Shivam |
| Ночная | 22:00–06:00 | Concierge A (sleeping duty) | Shivam → Roman |

**Sleeping duty** — человек спит, но телефон на громкой. При вызове обязан ответить в 3 мин. Надбавка к ставке.

**Минимум:** 2 concierge в Индии + Шивам как third-level escalation. Не наоборот.

## База данных для дежурного (must have)

Без этого дежурный беспомощен:

- **Госпитали по городам** (с рейтингом, адресом, английским персоналом)
- **Полиция туристическая** по штатам
- **Консульство РФ в Индии** (Дели, Мумбаи, Ченнаи, Калькутта)
- **Адвокаты** (русскоязычные или с переводчиком)
- **Страховые** (топ-3 компании с инструкцией)
- **Такси/трансфер экстренный** по городам (проверенные, не рандомные)
- **Отели резервные** (если клиента выселили или не пустили) по ключевым маршрутам

Это живая база в системе, не PDF. Обновляется командой.

## SOS и платформа (фаза 4)

Когда SOS станет сервисом для других операторов:

- Оператор настраивает своих дежурных или подключает наш пул
- Playbooks адаптируются под страну (Марокко → другие госпитали, другая полиция)
- SLA и отчётность — в личном кабинете оператора
- Billing: оператор платит за активированные SOS-тикеты или фиксированную ставку дежурства

**Это полноценный B2B продукт**, который продаётся отдельно от остальной платформы.
