## Кому 2FA обязателен

| Роль | 2FA | Обоснование |
|---|---|---|
| **admin** (Roman, Shivam) | ✅ обязательно | Доступ ко всему, включая ПДн всех клиентов |
| **finance** | ✅ обязательно | Платежи, выплаты, финансовые данные |
| **manager / sales** | ✅ обязательно | Видят паспорта клиентов |
| **concierge** | ✅ обязательно | Видят активные поездки, паспорта, экстренные контакты |
| **гид** | ⚠️ опционально (рекомендуется) | Видит только своих текущих клиентов в день смены |
| **клиент** | ⚠️ опционально (рекомендуется) | Доступ к своему дашборду |

В фазе 3 — обязательность 2FA для всех team ролей не обсуждается.

## Методы

### Основной: TOTP (Google Authenticator, Authy, 1Password)

- Стандарт RFC 6238
- 6-значный код, обновляется каждые 30 секунд
- Без зависимости от SMS / push-инфраструктуры

### Резервный: одноразовые коды (recovery codes)

- 10 кодов выдаются при включении 2FA
- Каждый код используется один раз
- Хранятся клиентом отдельно от устройства

### Альтернатива (для будущей фазы): WebAuthn / passkeys

Биометрия + аппаратный ключ. Сложнее в onboarding, но безопаснее.

## Включение 2FA при онбординге сотрудника

1. Создание аккаунта sales/concierge/etc → email с приглашением
2. Установка пароля
3. **Перед первым входом — обязательная настройка 2FA:**
   - QR-код для Authenticator
   - 10 recovery-кодов с обязательным подтверждением «сохранил»
4. Только после этого аккаунт активен

## Восстановление доступа

### Утрачено приложение, есть recovery-коды

- Используется один recovery-код, доступ восстановлен
- Recovery-код вычеркнут
- Уведомление admin (Roman) о использовании recovery

### Утрачено и приложение, и recovery-коды

Проверка личности:
1. Видеозвонок с admin (Roman или Shivam, не оба сразу)
2. Подтверждение по корпоративному email + личному
3. (для team) подтверждение коллегой
4. Сброс 2FA через admin-консоль
5. Принудительная новая настройка при следующем входе
6. Логирование инцидента в audit-svc

**Срок восстановления:** до 24 часов. Если бизнес-критично — допустимо faster-track с двумя уровнями подтверждения.

## Реализация

`identity-svc`:

- Endpoint `POST /v1/auth/2fa/setup` — генерирует TOTP secret + QR
- Endpoint `POST /v1/auth/2fa/verify` — подтверждение секрета
- Endpoint `POST /v1/auth/2fa/recovery-codes` — выдача / регенерация
- Поле в БД `users.totp_secret` — зашифровано (см. [`SECRETS.md`](./SECRETS.md))
- Поле `users.recovery_codes` — bcrypt хеши (как пароли)

Каждый login после ввода пароля → запрос TOTP кода. JWT выдаётся только после.

## Метрики

В audit:
- `auth.2fa.success.rate` — % успешных входов с 2FA
- `auth.2fa.recovery.used` — счётчик recovery-кодов
- `auth.2fa.disabled.attempts` — попытки отключить 2FA (тревога)
